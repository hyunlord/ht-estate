# R2 [SPIKE] — 통합 후 조인 recall 회귀 확인

**판정: 회귀 아님.** 조인 로직은 T0-4c와 정확히 일치. C3 라이브의 45%는 **202504 달 구성 차이**(구조적 천장↑)이며 로직 손상이 아니다. C3 리턴팩의 "45%≈68.5% 일치" 표현은 부정확했다(서로 다른 달 비교) — 본 측정이 정정한다. **Phase 1 진입 OK.**

## 동기
C2 통합 + C3 재적재 후 조인 recall이 T0-4c(이름 60.3% / 이름+지번 68.5%, 강남 202602-04)와 일치하는지 확인. C3 라이브가 202504에서 matched 45%로 나와, **로직 회귀인지 달 차이인지** 측정으로 판정.

## 실행 1 — 강남 11680 · 202602/202603/202604 (T0-4c 기간)
```
단지(K-apt) 232 · 거래(MOLIT, dedup) 698
이름만        :  421 / 698  recall 60.3%
지번 포함     :  478 / 698  recall 68.5%
지번 회수분   : +57  (recall +8.2%)
--- 잔여 손실 분해(이름만 매칭 후) ---
  지번-회수 가능 : 57
  이름가드 차단  : 70
  지번 충돌      : 34
  구조적(천장)   : 116  (K-apt에 해당 지번 단지 없음 = 17%)
  미매치 합계    : 277
```
→ **T0-4c 수치(60.3% / 68.5% / +57) 정확 재현.** 조인 로직 정상.

## 실행 2 — 강남 11680 · 202504 (C3 라이브 달)
```
단지(K-apt) 232 · 거래(MOLIT, dedup) 113
이름만        :   50 / 113  recall 44.2%
지번 포함     :   51 / 113  recall 45.1%
지번 회수분   : +1  (recall +0.9%)
--- 잔여 손실 분해(이름만 매칭 후) ---
  지번-회수 가능 : 1
  이름가드 차단  : 28
  지번 충돌      : 1
  구조적(천장)   : 33  (K-apt에 해당 지번 단지 없음 = 29%)
  미매치 합계    : 63
```
→ **C3 라이브 matched 45% 정확 재현**(51/113 = 45.1%).

## 판정 근거
1. **회귀 아님** — 실행1이 T0-4c를 정확히 재현(60.3%/68.5%). 동일 로직이 동일 입력에 동일 결과 → C2 통합·C3 재적재가 조인을 손상시키지 않았다.
2. **45%는 달 차이** — 202504의 낮은 recall은 **구조적 천장**(K-apt에 해당 지번 단지 없음)이 29%로, 202602-04의 17%보다 크기 때문. 압구정·대치 등 구축/재건축 거래가 그 달 비중이 커 K-apt 등록 단지와 못 맺어진 것으로 보인다(천장은 알고리즘이 아닌 데이터 커버리지 한계).
3. **jibun 메커니즘 살아있음** — 두 달 모두 이름만 < 지번포함(60.3<68.5, 44.2<45.1). 202504는 회수 가능분 자체가 1건뿐이라 지번 기여가 작을 뿐, 메커니즘은 정상 동작.

## 권고
- **Phase 1 진입 OK.** 조인은 회귀 없음.
- recall은 달마다 변동하며 주 동인은 **구조적 천장(K-apt 단지 커버리지)** — 알고리즘 한계가 아니라 데이터 한계. 단일 달의 낮은 수치를 "회귀"로 오인하지 말 것(여러 달 합산으로 보는 게 안정적).
- (정정) 리턴팩에 라이브 recall을 보고할 땐 **측정한 달을 명시**하고 다른 달 수치와 직접 비교하지 말 것.
