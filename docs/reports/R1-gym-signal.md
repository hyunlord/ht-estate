## 0. 권고 한 줄 (TL;DR)

**헬스장은 K-apt로는 불가능하다(표본 17단지 true-positive 0/17).** 따라서 설계 §5의 **Tier-1 결정론 `has_gym`(welfareFacility 파싱) 전제는 폐기**하고, 헬스장을 **Tier-2 lazy enrichment(§6)** 로 이전한다. 후보(~20)에 한해 **① 단지 공식홈·입주자모집공고(권위·합법 1순위) → ② 호갱노노·KB부동산·부동산뱅크 단지 시설(반정형, 개인리서치·비재배포 한정) → ③ 웹검색+LLM 문맥추출(인근 상업헬스장과 구별 필수)** 을 종합한다. `has_gym`은 **hard filter가 아니라 `(값, source_type, source_url, fetched_at, confidence)` 동반 soft 속성**으로 두고, 신호가 없으면 `NO`가 아니라 **`UNKNOWN`(랭킹 강등하되 제외 금지, §7)** 으로 처리한다.

---

## 1. 배경

- T0-2에서 K-apt `welfareFacility`에 헬스장 데이터가 사실상 없음을 발견(강남·송파·서초 ~180단지 0건, 역삼자이 등 프리미엄도 0).
- 설계 §5는 `has_gym`을 **Tier-1 결정론 필드**(K-apt 기본정보 → `amenities_raw` 파싱)로, §10 Phase 0는 이를 "네이버 불가" 플래그십 필터로 못박았다. **이 전제가 성립하는지**를 커밋 전에 검증하는 것이 이 스파이크의 목적.
- 이 리포트는 **어느 소스가 헬스장 유무를 신뢰성 있게 주는가**를 표본 측정 + 소스별 법적/실현가능성으로 답하고 Phase 1 방식을 권고한다.

## 2. 방법론

### 2.1 표본 (17단지, 강남권+비강남 / 신축·구축 대조 / 유명·무명 혼합)
실 단지코드는 `kapt.list_complexes(sigungu=…)` 라이브 조회로 확보(강남구 232·서초 199·송파 178·마포 140·노원 230·인천부평 159단지에서 선별). 유명 단지는 이름 부분일치, 비강남 long-tail은 리스트 스프레드 후보를 기본정보로 **소형(<300세대)·구축(<2005) 우선** 자동 선별.

### 2.2 Ground-truth(GT) 라벨 — 순환성 회피
- GT = **단지 내 입주민 전용 피트니스/헬스장/주민운동시설** 보유 여부. **인근 상업 헬스장·구민체육센터는 GT에서 NO**(이 구별이 측정의 핵심).
- GT는 **권위 소스**(단지 공식홈·입주자모집공고·관공서 웹진·언론·입대의/약관)로 잡아 **평가 대상인 "naive 웹검색 신호"와 분리**했다.
- 확정 불가 단지는 **`UNKNOWN`** 으로 남겼다(소거증명 불가 — "신호 없음 ≠ 헬스장 없음"). 이 자체가 long-tail finding.

### 2.3 소스별 평가기준
- **K-apt**: basis(31필드)+detail(36필드) **전 문자열 필드**를 헬스장 토큰(`헬스/피트니스/휘트니스/주민운동/체력단련/스포츠센터/운동시설/fitness/gym/GX/…`)으로 스캔, 매칭 필드·문자열 기록(감사 가능).
- **웹검색**: 단지당 naive 쿼리 1회(`{단지명/동} 헬스장 피트니스 커뮤니티 주민운동시설`), firecrawl 검색(글로벌·본문추출). 신호를 in-complex/인근/무신호로 분류, GT와 대조해 적중·오탐·누락 산정. **강남 vs 비강남 분리**.
- **서드파티(네이버/호갱노노/아실/공식홈)**: 의뢰서 제약에 따라 **문서 리서치만**(자동 fetch 0). 다윈중개 판례·DB권·ToS·소유관계 근거로 법적/실현가능성 판정.

## 3. 소스 A — K-apt (천장 재확인) → 헬스장 신호 **0/17 (true-positive)**

| 지표 | 결과 |
|---|---|
| 토큰 매칭 단지 | **1/17** (갈산대동2차) — 그러나 **오탐** |
| 진짜 헬스장 적중(true-positive) | **0/17** (강남권 0/8, 비강남 0/9) |
| 시설 관련 필드 | `welfareFacility`, `convenientFacility`, `educationFacility` — 그 외 **항목화된 부대시설 필드 없음** |

**(1) `welfareFacility`는 고정 카테고리 체크리스트** — 헬스장을 항목화하지 않는다. 관측된 값 집합: `{관리사무소, 노인정, 보육시설, 문고, 주민공동시설, 커뮤니티공간, 어린이놀이터, 휴게시설, 유치원, 자전거보관소, 기타}`. 헬스장은 "주민공동시설/커뮤니티공간/기타" 버킷에 흡수되어 **개별 명시가 없다**.
- 헬리오시티(9,510세대, 대형 커뮤니티 피트니스·수영장 확실): `관리사무소, 노인정, 문고, 주민공동시설, 어린이놀이터, 휴게시설, 유치원, 커뮤니티공간, 자전거보관소, 기타` → 헬스장 무표기.
- 디에이치 아너힐즈(스포츠센터 확실): `…주민공동시설, 어린이놀이터, 휴게시설, 커뮤니티공간…` → 무표기.
- 역삼자이: `…어린이놀이터, 커뮤니티공간, 자전거보관소` → "주민공동시설"조차 없음.

**(2) `convenientFacility`/`educationFacility`는 단지 시설이 아니라 인근 POI** — 헬스장 신호로 부적합. 게다가 결측 빈번: 역삼자이 `관공서() 병원()`, 아크로리버파크/신반포자이/부평LH1 `''`(빈값).

**(3) 유일 토큰 매칭은 오탐.** 갈산대동2차의 `운동시설`은 `convenientFacility = "…기타(굴포천 자전거도로 및 산책로 운동시설)"` — **인근 하천 산책로 야외운동기구**다. 동시에 이 단지는 실제로 단지 내 휘트니스클럽을 운영(§4 참조)하지만 `welfareFacility`엔 빠져 있다. 즉 **K-apt는 오탐과 누락을 동시에** 낸다.

> **결론: K-apt는 헬스장 신호로 구조적으로 부적합.** welfareFacility는 시설 유무가 아닌 카테고리 분류이고, 헬스장을 항목화하는 필드·엔드포인트가 두 V4 응답(67필드) 어디에도 없다. 설계 §5의 Tier-1 `has_gym` 전제는 성립하지 않는다.

### 4.2 권역별 요약 (long-tail 신뢰도)

| 권역 | 표본 | 고신뢰 적중 | 오탐/토큰실패 | 묻힘(정답 존재하나 noise) | 무신호 |
|---|---|---|---|---|---|
| **강남권** | 8 | **4** (디에이치·아크로·헬리오·래대팰, 공식홈/언론/관공서로 고정밀) | 2 (은마 오탐, 잠실엘스 토큰오탐) | 2 (신반포·역삼) | 0 |
| **비강남** | 9 | **0** | 1 (산곡 오탐위험) | **5** (마포·갈산·부개·공릉·월계 — 전부 반정형 1건씩, noise에 묻힘) | **3~4** (상계·하계·부평LH) |

**핵심 패턴**
1. **강남 프리미엄 신축은 웹이 강하다** — 공식홈(`acroriverpark.kr`·`palaceapt.co.kr`)·관공서 웹진·언론·KB부동산 등 **권위/반정형 소스가 고정밀**로 in-complex 헬스장을 확정. 단, naive 토큰 매칭은 위험(잠실엘스: GT=NO인데 "길건너 헬스장"에 오탐, 은마: 전부 인근 상업헬스장).
2. **비강남 long-tail은 신뢰 불가** — 정답 신호가 있어도(5/9) **항상 noise에 묻히고 반정형 단발 소스에 의존**(부동산뱅크·업체디렉터리·hogangnono 댓글 1건). 무신호 3~4/9. **자동 추출하면 오탐 또는 무신호**.
3. **오탐 메커니즘이 지배적**: ⓐ 인근 상업 헬스장(은마→스포애니, 산곡→펀휘트니스), ⓑ 구민/공공 체육센터(공릉→공릉구민체육센터), ⓒ 브랜드 사칭(마포→"짐박스 마래푸점"), ⓓ 브랜드 충돌(신반포자이→메이플자이). → **naive 토큰 추출 금지, in-complex 판별 LLM 문맥추출 필수**.
4. **GT 자체가 반례** — 잠실엘스(2008·5,678세대)는 헬스장 **없음**, 1980~90년대 비강남 구축(갈산대동2차·부개대동)은 헬스장 **있음**. → **세대수/연식으로 has_gym을 추정할 수 없다**(K-apt 파생 대체 불가).

## 5. 소스 C — 서드파티 플랫폼 (문서 리서치 / 자동 fetch 0)

> 자동 fetch는 검색엔진 인덱스 스니펫만 사용했고 네이버/호갱노노/아실 **매물·단지 페이지 직접 fetch는 0건**. 아래 법적 판단은 라이브 검색으로 1차 확인한 출처에 근거.

### 5.1 네이버 부동산 — `not-viable`
- **데이터**: 단지정보 > "단지 부대시설"에 편의·복리시설이 자유서술로 노출(헬스장은 그 안의 토큰일 가능성 높음). 공개 API 없음, JS 렌더 + 강한 봇차단.
- **법적(다윈중개 판례 — 확정)**: 다윈프로퍼티가 2021.2~2025.3 네이버 부동산 매물정보를 복제·전송 → 법원이 **네이버를 데이터베이스 제작자로 인정하고 DB제작자 권리 침해**로 보아 **8,000만원 배상 + 다윈 서버의 네이버 DB 삭제** 명령, **양측 상고하지 않아 확정**, 다윈중개는 **2026.2 서비스 종료**. (네이버 상표 사용에 대한 부정경쟁방지법 제2조1호(나)목 주장은 링크 안내 목적으로 보아 **기각**.) 출처: [리걸타임즈](https://www.legaltimes.co.kr/news/articleView.html?idxno=94027) · [전자신문](https://www.etnews.com/20241007000224) · [한국경제](https://www.hankyung.com/article/2026010406071) · [클리앙(2심 승소)](https://www.clien.net/service/board/news/19120710).
- **판단**: 사업자 간 **상업적 대량 복제** 사안 → 개인·비영리·소량은 DB권 "상당부분 복제" 요건 충족이 어렵지만, **강한 봇차단 + 판례 본체 사정권**으로 개인용도 비권장, 상업화는 명백한 적색선.

### 5.2 호갱노노 — `not-viable`(개인 참조는 가능, 스크레이핑 불가)
- **데이터**: 단지 단위 편의시설·커뮤니티(헬스장 포함)·실거래·**입주민 게시판**이 풍부 — 본 조사에서 in-complex 정답을 가장 자주 제공(잠실엘스 "없음", 디에이치 "있음", 공릉우성 "107동 헬스클럽", 신반포자이·마포래미안 입주민 Q&A).
- **소유/법적**: **직방이 2018년 지분 100%를 약 230억원에 인수한 완전자회사**([톱데일리](https://www.topdaily.kr/articles/91271) · [더벨](https://www.thebell.co.kr/front/newsview.asp?key=201804120100024730001522)). 직방 계열 = 다윈 판례 사정권 + 분쟁 적극성. 앱월·ToS 자동수집 금지 리스크.
- **판단**: **개인 리서치의 참조(사람이 읽는)** 용도는 가용하나 **자동 스크레이핑/파이프라인·재배포·상업화는 금지**.

### 5.3 아실 — `viable-but-risky`
- 단지·실거래·재건축 강점, 단지 상세에 부대시설 포함 가능. 공개 API 없음, JS·내부 비공식 API 의존. ToS/robots 현행 문구는 **라이브 미확인**.
- 핵심 데이터의 공공(실거래) 비중이 있어 "순수 사실"은 비보호이나 선택·배열·가공(DB 구조)은 보호 주장 여지. 개인 소량은 리스크 낮음, 상업·대량은 적색선.

### 5.4 단지 공식홈 / 입주자모집공고 / 관리사무소·입대의 게시 — `viable-source` (권위 1순위)
- **데이터**: 입주자모집공고는 부대복리시설(헬스장·GX·골프연습장)을 **법정 고지**. 공식홈은 시설/예약 페이지를 정형 제공 — 본 조사에서 **아크로리버파크(`acroriverpark.kr` 헬스장 예약)·래미안대치팰리스(`palaceapt.co.kr` 약관)** 가 실증.
- **약점**: 커버리지 = **프리미엄/대형 위주**(long-tail엔 거의 없음), 기계가독성 나쁨(공고는 PDF/이미지, 공식홈은 단지별 산재·분양 후 폐쇄 잦음).
- **법적**: 충돌 거의 없음(공익 고지 + 단지 자기 정보). **가장 청정.**
- **판단**: **후보(~20) lazy 출처 1순위**(있을 때), 전국 1차 적재용으론 부적합.

## 7. Phase 1 권고

### 7.1 방식 — 헬스장을 Tier-2 lazy enrichment로 (복수 소스 종합)
설계 §6의 lazy read-through(후보 한정)로 이전. 후보(~20)당 우선순위:
1. **단지 공식홈 + 입주자모집공고**(권위·합법 1순위) — 있으면 confidence 최상. PDF/이미지는 OCR+LLM.
2. **호갱노노·KB부동산·부동산뱅크 단지 시설/커뮤니티**(반정형, in-complex 정답률 높음) — **개인 리서치 참조·비재배포 한정, 자동 스크레이핑·상업화 금지**.
3. **웹검색(블로그·카페·유튜브·언론) + LLM 문맥추출** — **naive 토큰 매칭 금지**. 추출기는 "단지 내 입주민 시설"과 "인근 상업 헬스장/구민체육센터"를 **명시적으로 구별**해야 함.

### 7.2 데이터 모델 — has_gym은 hard filter가 아니라 soft+provenance
- 원칙 1(provenance)에 따라 `(값, source_type, source_url, fetched_at, confidence)`로 적재.
- **confidence 등급**: 공식홈/입주공고/관공서 = high · 호갱노노/KB/부동산뱅크 단지 게시 = medium · 단발 블로그/카페(인근 모호) = low+`확인 권장`.
- **신호 없음 = `UNKNOWN`(≠ NO)**. §7 랭킹 규칙대로 `required`라도 **강등하되 제외 금지**(소거증명 불가).

### 7.3 설계 수정사항 (위반 아님 — 전제 정정)
- **§5/§10**: "Tier-1 결정론 `has_gym`(K-apt) = 플래그십 hard filter" → **불성립**. 헬스장은 Phase 0 hard filter에서 빼고 Phase 1 enrichment soft 속성으로 재배치.
- **§4 데이터 모델**: `complex.has_gym`(결정론 컬럼)을 `enrichment`(단지×속성 다중행) 경로로. provenance 필드 유지.
- 세대수/연식 휴리스틱으로 has_gym을 추정하지 말 것(잠실엘스 반례).

### 7.4 한계 / 검증 부채
- 표본 17(설계상 ~18). long-tail GT 4건은 `UNKNOWN`(소거증명 불가) — finding이지 누락 아님.
- 웹검색 신호는 단지당 naive 쿼리 1회 기준. 실제 enrichment는 다중 쿼리·소스로 recall 상승 여지(정밀도 부담과 trade-off).
- **미확인(사람 재확인 권장)**: 네이버 "단지 부대시설" 필드 정규화 형태, 아실 ToS/robots 현행 문구, 다윈 판례 심급 체계(1심 법원·항소심 관할). 본 리포트는 판결 **핵심(DB제작자 권리 침해·확정·서비스종료)** 만 다중출처로 확정.
